Лабораторная работа №2
Обеспечение отказоустойчивости распределённой системы CrackHash

Технологический стек (в задании): Java, Spring Boot, Docker Compose

Источник (Google Docs):
https://docs.google.com/document/d/1Au89bzSHsfz9POnSHyO8iNv-zRoA9kCADBIZvwkrRs8/

-----------------------------------------------------------------------------
1. Введение

Усилить отказоустойчивость системы CrackHash. Главная цель — гарантировать
сохранность данных и успешную обработку пользовательских запросов при
работоспособности менеджера. Система должна корректно переживать остановку и
перезапуск компонентов (менеджер, диспетчер, воркеры, БД, брокер) без потери
данных или невыполненных задач.

-----------------------------------------------------------------------------
2. Архитектура системы

2.1 Менеджер (Manager)
  - Принимает пользовательские HTTP-запросы.
  - Проверяет идемпотентность по параметрам запроса.
  - Сохраняет запрос и состояние в MongoDB.
  - Возвращает requestId клиенту только после успешного сохранения и репликации.
  - Читает статус и результат из MongoDB.
  - Не взаимодействует с RabbitMQ напрямую.

2.2 Диспетчер задач (Task Dispatcher)
  - Отслеживает новые и изменённые записи в MongoDB.
  - Нарезает запрос на части (parts), создаёт записи подзадач в MongoDB.
  - Публикует части в очередь воркеров, принимает результаты.
  - Отмечает части выполненными, повторяет при таймауте или сбое воркера.
  - Завершает весь запрос: статус READY, ERROR или CANCELLED.
  - При недоступности RabbitMQ сохраняет задания в MongoDB и отправляет после
    восстановления очереди.

2.3 Воркеры (Worker)
  - Получают части заданий из RabbitMQ.
  - Перебор слов в диапазоне, вычисление хэша, результат через RabbitMQ.
  - Подтверждают сообщения только после корректной обработки.
  - Могут повторно обработать часть при повторной доставке после сбоя.

2.4 MongoDB Replica Set
  - Хранилище запросов, частей, результатов, служебных записей.
  - Развёртывание: replica set — 1 primary и 2 secondary.
  - Менеджер подтверждает приём запроса только при write concern majority.
  - Работа после смены primary без ручной правки логики приложения.

2.5 RabbitMQ
  - Обмен между диспетчером и воркерами.
  - Очереди и сообщения — персистентные.
  - Достаточно direct exchange.
  - При недоступности потребителя сообщения остаются в очереди до восстановления.

-----------------------------------------------------------------------------
3. Функциональные требования

3.1 Сохранность пользовательского запроса
  - После POST /api/hash/crack — запись в MongoDB.
  - До успешной записи и репликации нельзя отвечать клиенту, что задача принята.
  - Идемпотентность при совпадении параметров: тот же requestId или готовый результат.

3.2 Сохранность при отказе диспетчера
  - Все сведения о запросе, частях, промежуточных результатах и статусах — в MongoDB.
  - После остановки диспетчера данные не теряются.
  - После рестарта диспетчер продолжает по данным MongoDB.
  - Ответы воркеров, не считанные диспетчером, не теряются.

3.3 Сохранность при отказе воркера
  - При остановке воркера часть не считается завершённой без ack.
  - Без подтверждения RabbitMQ переотправляет сообщение другому воркеру.
  - Диспетчер учитывает повторную обработку части и не дублирует результат.

3.4 Сохранность при отказе очереди
  - При недоступности RabbitMQ задания не теряются: MongoDB, статусы NEW или
    QUEUED_PENDING (как в спецификации).
  - После восстановления RabbitMQ — повторная отправка накопившихся заданий.
  - После рестарта RabbitMQ персистентные сообщения не теряются.

3.5 Сохранность при отказе MongoDB primary
  - Переизбрание новой primary.
  - Система продолжает работу с replica set без ручного изменения логики.
  - Подтверждение клиенту при write concern majority.

-----------------------------------------------------------------------------
4. API менеджера

(Совместимо с лабораторной 1 по контракту.)

4.1 POST /api/hash/crack
  Body: hash, maxLength, algorithm, alphabet (дефолтный алфавит a-z0-9).

4.2 Ответ
  requestId, estimatedCombinations; состояние сохраняется (в задании — в MongoDB,
  статус IN_PROGRESS после надёжной записи).

4.3 GET /api/hash/status?requestId=...
  IN_PROGRESS, READY + data, ERROR + error, CANCELLED.

4.4 DELETE /api/hash/crack?requestId=...
  CANCELLED, уведомление воркеров, освобождение ресурсов.

4.5 GET /api/metrics
  totalTasks, activeTasks, completedTasks, avgExecutionTime
  (в спецификации — средняя скорость перебора, слов/сек).

-----------------------------------------------------------------------------
5. Распределение задач

5.1 Pull-модель (в отличие от лаб. 1)
  - Менеджер пишет запрос в MongoDB.
  - Диспетчер видит новый IN_PROGRESS, считает partCount, создаёт части в MongoDB.
  - Сообщения по частям в RabbitMQ.
  - Воркеры забирают части, считают, результат в отдельную очередь результатов.
  - Диспетчер читает результат, фиксирует в MongoDB, закрывает часть.
  - Все части готовы — обновление общего статуса запроса в requests.

-----------------------------------------------------------------------------
7. Развёртывание

7.1 Docker Compose
  Разное число воркеров (1, 4, 8) или масштабирование, например:
  docker compose up --scale worker=4
  Переменная WORKERS_COUNT (как в типичной конфигурации задания).

7.2 Сеть
  - Клиент — только HTTP к менеджеру.
  - Менеджер и диспетчер — MongoDB по внутренней сети compose.
  - Диспетчер и воркеры — RabbitMQ внутри compose.
  - Имена сервисов compose как домены.
